什麼是GitHub Enterprise?

GitHub Enterprise 是企業版的程式碼代管與協作平台。對 Data Team 而言，它不只是存放程式碼的地方，也是日常開發、協作與版本控管的核心工作平台。

  • 集中化版本控制 安全儲存與管理 ETL 腳本、資料處理流程、分析工具與專案設定檔。
  • 團隊協作與審查 透過 Pull Request、Review 與 Issue 管理，讓多人協作更有秩序。
  • 權限與組織治理 可搭配 SSO 與組織權限控管，避免未授權人員存取專案內容。
  • 自動化流程 可利用 GitHub Actions 建立測試、部署與排程化的工作流程。
  • Copilot 輔助開發 在企業授權環境中，於 VS Code 中取得程式碼補全與對話式協助。

初次使用：連接 GitHub Enterprise 與 VS Code Copilot

第一次使用時，建議先完成帳號登入、組織授權與 Copilot 狀態確認，再開始開發工作。

  1. 安裝擴充套件：開啟 VS Code，進入左側「Extensions (擴充功能)」，搜尋並安裝 GitHub Copilot 與 GitHub Copilot Chat。
  2. 登入公司核發的 GitHub 帳號：
    • 點擊 VS Code 左下角的「帳戶 (Accounts)」圖示。
    • 依公司使用情境選擇登入 GitHub 或 GitHub Enterprise 的選項，並使用已核發的帳號完成登入。
  3. 完成組織授權或 SSO 驗證：瀏覽器開啟後，請依公司要求完成登入、授權或單一登入驗證。
  4. 若系統要求輸入 Enterprise 網址：請填入組內提供的 GitHub Enterprise 網址或 Server URI，並以實際配發資訊為準。
  5. 確認 Copilot 狀態：驗證成功後，VS Code 右下角會顯示 Copilot 圖示，且不應出現停用或授權失敗的狀態。
  6. 進行簡單測試：開啟任一程式檔案，輸入幾行註解或函式名稱，確認Copilot是否順利產生回應。

建議：若已成功登入，但看不到 Copilot 建議或對話功能，通常可先嘗試重新整理 VS Code 視窗，或重新登入帳號以同步授權狀態。

Copilot 「+」選單教學

當您點擊對話框旁的「+」圖標時，您可以手動指定 Copilot 參考特定的資訊來源（Attachments），這能有效避免 AI 產生幻覺或給出不相關的建議。
以下是選單中各項功能的詳細說明：

📄 Open Editors

讓 AI 參考目前您已經開啟的所有分頁檔案。

適用情境：當您正在進行多檔案開發，且各檔案間有邏輯關連時使用。

📁 Files & Folders...

從整個專案資料夾中，指定特定的檔案或資料夾作為參考。

適用情境：當您需要 AI 讀取尚未開啟的數據定義檔 (Schema) 或設定檔時。

🔖 Instructions...

讓 AI 參考特定的自訂指令文件。

適用情境：當團隊有統一的 Coding Style 或資料處理規範文件時。

🔀 Source Control...

參考目前的 Git 變更狀態或 Diff 內容。

適用情境：當您想請 AI 幫忙寫 Commit Message，或檢查剛改動的程式碼有無漏洞時。

❌ Problems...

將 VS Code 偵測到的語法錯誤或警告 (Errors/Warnings) 餵給 AI。

適用情境：快速 Debug：直接讓 AI 看到底層報錯訊息並尋求解決方案。

📦 Symbols...

搜尋並參考專案中的特定變數、函式或類別 (Classes)。

適用情境：當專案很大，您只想針對某個特定函式的邏輯進行優化時。

💬 Sessions...

參考過往的對話紀錄。

適用情境：需要延續之前的討論主題，或從之前的解法中延伸新功能時。

🛠️ Tools...

允許 Copilot 使用外部輔助工具（如 Terminal 執行結果）。

適用情境：當需要 AI 根據執行環境的輸出結果進行分析時。

基本語法：Push 與 Pull

對團隊協作而言，最常用的 Git 操作通常是 Pull 與 Push。理解這兩個動作，有助於避免版本衝突並保持專案同步。

Pull (拉取)

定義：將 GitHub Enterprise 遠端儲存庫 (Remote Repository) 中最新的程式碼與變更，下載並合併到您本機的開發環境中。

使用時機：建議在每天開始工作前，或在準備修改共用檔案前先執行 Pull，以降低與其他同仁衝突的機率。

git pull origin main

註：實際分支名稱不一定是 main，請依專案使用的預設分支為準。

Push (推送)

定義：將您本機端已經 Commit (提交) 的變更，上傳至 GitHub Enterprise 的遠端儲存庫。

使用時機：當您完成一段可驗證的修改，並已於本機測試與 Commit 後，即可 Push 到遠端儲存庫與團隊共享。

git push origin main

註：在 VS Code 中，完成 Commit 後，通常可透過 "Sync Changes" 一併完成 Push 與同步動作。

實務建議：開始工作前先 Pull；完成一段明確修改後先 Commit，再 Push。這樣較容易追蹤版本，也方便同仁閱讀與回溯變更內容。
