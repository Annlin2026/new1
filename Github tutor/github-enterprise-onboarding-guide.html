用自然語言完成 GitHub Enterprise 與 Copilot Chat 的日常工作

這份頁面直接根據原始文字說明整理，目標是讓第一次接觸 GitHub Enterprise、Git 與 VS Code 內 GitHub Copilot Chat 的同仁，能迅速掌握操作順序與實務用法。

適用對象：組內新手同仁
核心工具：GitHub Enterprise / Git / VS Code Copilot Chat
目標：理解、生成、提交、推送

這份說明涵蓋什麼

  • 查看資料夾與檔案內容
  • 摘要既有文件
  • 依照現有檔案產生新版本
  • 生成對外介紹文件
  • 將變更提交並推送到遠端 Repository
目的 不要求一開始就熟記所有 Git 指令，而是先學會把需求講清楚，讓工具協助完成實際任務。

一、你可以用這個工具做什麼

Capabilities

在 VS Code 的 Copilot Chat 中，你可以直接用自然語言要求系統協助處理檔案與 Git 工作。下面把用途、提問方式與使用情境整理成可直接看的卡片。

1

查看目前資料夾內容

快速了解目前工作區有哪些檔案，先確認是否已存在目標檔案。

範例提問 
現在檔案夾裡有什麼?
2

摘要文件內容

快速掌握 HTML、文字檔或教學文件重點，適合接手別人建立的教材或專案。

範例提問 
摘述檔案重點
3

依既有內容生成新版本

從既有版本延伸出新版本，精簡內容、改寫語氣、調整結構，適合教學文件、簡報稿與對外說明稿。

範例提問 
請依照兩份HTML修改出一份V3版本，限制在3頁以內
4

生成對外介紹文件

把偏內部說明轉成正式、對外可用的介紹文字，可作為公告、README 或對外提案基礎。

範例提問 
生成一版對外介紹
5

協助推送到 GitHub Enterprise

協助完成 Git 初始化、commit 與 push，特別適合第一次不熟 Git 指令的同仁。

範例提問 
push please
範例提問 
把這個專案push到我的github remote repo

二、這次實際用到的工作流程

Workflow

以下是這次對話實際完成的順序，可直接當成第一次帶人的標準範例。

1

檢查工作區

確認資料夾內有哪些 HTML 檔案，了解目前教材的版本結構。

2

閱讀並摘要既有文件

讀取 claude-code-tutorial.html，整理出內容主題、結構與重點段落。

3

生成新版本檔案

以既有教學頁面為基礎建立 claude-code-tutorial-v3.html，將長篇內容濃縮成較適合列印與速讀的版本。

4

建立補充文字文件

建立 folder-summary.txt 與 public-introduction.txt，補足摘要與對外版本。

5

提交到 GitHub

初始化 Git Repository、新增 remote、commit 檔案，最後 push 到 main branch。

三、第一次使用 GitHub Enterprise 前的準備

Preparation

先確認你有以下權限

  • 已取得 GitHub Enterprise 帳號
  • 已被加入對應的 Organization 或 Repository
  • 有該 Repository 的讀寫權限

本機需安裝 Git

Windows 建議先安裝 Git for Windows。安裝完成後可在終端機執行以下指令：

檢查版本
git --version
建議先設定 Git 使用者資訊 這一步很重要，否則 commit 可能會顯示成自動偵測名稱。
建議執行
git config --global user.name "你的姓名"
git config --global user.email "你的公司信箱"

四、Git 與 GitHub Enterprise 基本操作

Git Basics

這一段把第一次最常用的 Git 指令分成八個步驟，照順序做就能完成從初始化到推送。

1. 檢查目前狀態

指令
git status --short --branch

用途：看目前在哪個 branch，以及哪些檔案尚未提交。

2. 初始化 Repository

指令
git init

適用情境：當資料夾還不是 Git Repository 時。若出現 fatal: not a git repository，代表需要先初始化。

3. 加入檔案到暫存區

指令
git add -A
git add -- "Tutorials/檔名"

可一次加入全部檔案，也可以只加入指定檔案。

4. 建立 commit

範例
git commit -m "說明這次修改內容"
git commit -m "Add condensed Claude Code tutorial v3"

commit message 建議具體描述修改內容，方便後續追蹤。

5. 設定遠端 Repository

指令
git remote add origin https://github.example.com/org/repo.git

公司若使用 GitHub Enterprise Cloud 或 Enterprise Server，網址可能不是 github.com，而是公司自己的網域。

6. 將主要分支改名為 main

指令
git branch -M main

讓本地主要分支名稱與遠端慣例一致。

7. 第一次 push 到遠端

指令
git push -u origin main

用途：將本地 main branch 推送到遠端，並建立本地與遠端的追蹤關係。

8. 後續再次推送

指令
git push origin main

之後只要有新 commit，使用這行就能再次同步。

五、第一次 push 時可能遇到的情況

Push Notes
瀏覽器登入驗證

你可能會看到類似訊息：

訊息範例
info: please complete authentication in your browser...

代表系統正在等你到瀏覽器完成登入或公司 SSO 驗證。完成後 push 會繼續進行。

作者資訊是自動偵測的

如果 commit 後看到系統提示你的 name / email 是自動推測，請盡快設定：

設定指令
git config --global user.name "你的姓名"
git config --global user.email "你的公司信箱"
LF / CRLF 警告

Windows 上可能看到：

警告內容
LF will be replaced by CRLF

這通常是換行格式提示，不一定是錯誤。若團隊已有既定規範，再依規範調整 Git line ending 設定。

七、建議的工作習慣

Habits
1

先看再改

先請 Copilot 摘要或說明檔案，再進入修改，比直接改更安全。

2

一次只處理一個明確需求

例如先做摘要、再做生成 v3、最後才做 push，可降低出錯率。

3

commit message 要清楚

建議 commit 訊息具體描述修改內容，方便後續追蹤。

4

push 前先看狀態

先執行 git status，確認沒有不想提交的檔案混進去。

5

對外文件與內部文件分開

像本次做法一樣，可將內部摘要與對外介紹分開維護，避免誤用。
