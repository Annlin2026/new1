## 一、什麼是大數據？

大數據（Big Data）指的是**傳統資料處理工具無法有效處理**的超大規模資料集合。

### 大數據的 5V 特性（必考！）

| V | 英文 | 中文 | 說明 |
|---|------|------|------|
| **V1** | Volume | 量大 | 資料量極大（TB、PB 等級） |
| **V2** | Velocity | 速快 | 資料產生與處理速度極快 |
| **V3** | Variety | 多樣 | 資料類型多元（文字、圖片、影片…） |
| **V4** | Veracity | 真實 | 資料品質與可信度 |
| **V5** | Value | 價值 | 從海量資料中萃取出有價值的資訊 |

> 💡 **口訣**：大數據就像大海 →「**量大、速快、多樣、真實、有價值**」

> ⚠️ 有些題目只考 3V（Volume, Velocity, Variety），有些考 5V，都要會！

## 二、結構化 vs 半結構化 vs 非結構化資料

| 類型 | 說明 | 範例 |
|------|------|------|
| **結構化資料** | 有固定格式，可存放在關聯式資料庫 | Excel 表格、SQL 資料庫、CSV |
| **半結構化資料** | 有一定結構但不嚴格 | JSON、XML、HTML、Log 檔 |
| **非結構化資料** | 沒有固定格式 | 圖片、影片、音訊、社群貼文、Email |

> 💡 大數據中，約 **80% 是非結構化資料**

## 三、Hadoop 生態系（重要！）

### Hadoop 核心元件

| 元件 | 功能 | 白話說明 |
|------|------|----------|
| **HDFS** | 分散式檔案系統 | 把大檔案切成小塊，分散存在多台電腦上 |
| **MapReduce** | 分散式運算框架 | 把大任務拆成小任務，分給多台電腦處理，最後合併結果 |
| **YARN** | 資源管理 | 負責分配 CPU、記憶體給各個任務 |

### HDFS 架構
```
                ┌──────────────┐
                │  NameNode    │ ← 管理員：知道資料存在哪
                │ （主節點）    │
                └──────┬───────┘
           ┌───────────┼───────────┐
     ┌─────┴─────┐ ┌───┴─────┐ ┌──┴──────┐
     │ DataNode 1│ │DataNode2│ │DataNode3│ ← 工人：實際存放資料
     │  資料塊A  │ │ 資料塊B │ │ 資料塊C │
     │  資料塊B  │ │ 資料塊C │ │ 資料塊A │ ← 每塊資料存 3 份（備份）
     └───────────┘ └─────────┘ └─────────┘
```

### MapReduce 運作流程
```
原始資料 → Map（拆解） → Shuffle（排序分組） → Reduce（合併） → 結果

範例：統計文章中每個字出現幾次
  Map:    "Hello World Hello" → (Hello,1)(World,1)(Hello,1)
  Shuffle: (Hello,[1,1])(World,[1])
  Reduce:  (Hello,2)(World,1)
```

## 五、Apache Spark（重要！）

### Spark vs MapReduce

| 比較項目 | MapReduce | Spark |
|----------|-----------|-------|
| 處理速度 | 慢（每步都寫入硬碟） | **快 10~100 倍**（記憶體運算） |
| 適用場景 | 批次處理 | 批次 + 即時串流 + 機器學習 |
| 程式語言 | Java 為主 | Scala, Python, Java, R |
| 易用性 | 較複雜 | 較簡單 |

### Spark 核心概念
- **RDD**（Resilient Distributed Dataset）：彈性分散式資料集，Spark 的基本資料單元
- **DataFrame**：類似 SQL 表格的結構化資料（和 pandas 的 DataFrame 概念類似）
- **Spark SQL**：用 SQL 語法查詢 Spark 資料
- **Spark Streaming**：即時串流處理
- **MLlib**：機器學習函式庫

## 六、ETL 流程（必考！）

```
   ┌─────────┐      ┌─────────────┐      ┌─────────┐
   │ Extract │  →   │  Transform  │  →   │  Load   │
   │  擷取   │      │    轉換     │      │  載入   │
   └─────────┘      └─────────────┘      └─────────┘
   從各種來源        清理、整合、          存入目標
   取出原始資料      轉換資料格式          資料倉儲
```

| 階段 | 說明 | 範例 |
|------|------|------|
| **E**xtract（擷取） | 從各種資料來源取出資料 | 從資料庫、API、檔案讀取資料 |
| **T**ransform（轉換） | 清洗、整理、轉換格式 | 去除重複、填補空值、格式統一 |
| **L**oad（載入） | 將處理好的資料存入目標系統 | 存入資料倉儲或資料湖 |

> 💡 **ELT**：先 Load 再 Transform，適合雲端環境（如 BigQuery、Snowflake）

## 七、資料倉儲 vs 資料湖

| 比較項目 | 資料倉儲 (Data Warehouse) | 資料湖 (Data Lake) |
|----------|---------------------------|---------------------|
| **資料類型** | 結構化資料 | 結構化 + 非結構化 |
| **處理方式** | 先整理再存入（Schema on Write） | 先存入再整理（Schema on Read） |
| **資料品質** | 高（已清洗整理） | 原始（未處理） |
| **使用者** | 商業分析師 | 資料科學家、工程師 |
| **查詢速度** | 快 | 看情況 |
| **成本** | 較高 | 較低 |
| **代表工具** | Amazon Redshift, BigQuery | Amazon S3, Azure Data Lake |

> 💡 **白話版**：
> - 資料倉儲 = 整理好的圖書館（書已分類上架）
> - 資料湖 = 什麼都丟進去的倉庫（要用時再找）

## 八、批次處理 vs 串流處理

| 比較項目 | 批次處理 (Batch) | 串流處理 (Streaming) |
|----------|------------------|----------------------|
| **處理時機** | 累積一批後統一處理 | 資料到達就立即處理 |
| **延遲** | 高（分鐘到小時） | 低（毫秒到秒） |
| **適用場景** | 月報表、歷史分析 | 即時監控、詐騙偵測 |
| **代表工具** | MapReduce, Spark Batch | Spark Streaming, Kafka, Flink |

## 九、NoSQL 資料庫

| 類型 | 說明 | 代表工具 |
|------|------|----------|
| **Key-Value** | 鍵值對，最簡單 | Redis, DynamoDB |
| **Document** | 文件型（JSON） | MongoDB, CouchDB |
| **Column-Family** | 列族型 | HBase, Cassandra |
| **Graph** | 圖形型（節點+關係） | Neo4j |

### SQL vs NoSQL

| 比較 | SQL（關聯式） | NoSQL（非關聯式） |
|------|---------------|-------------------|
| 結構 | 固定表格結構 | 靈活結構 |
| 擴展 | 垂直擴展（加強單機） | 水平擴展（加更多機器） |
| 交易 | 支援 ACID | 部分支援 |
| 適用 | 結構化資料、複雜查詢 | 大量非結構化資料 |
